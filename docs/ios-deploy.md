# iOS deploy pipeline

Kyuhachi builds iOS on a **self-owned GitHub Actions macOS runner with fastlane**
and uploads to **TestFlight** (internal testers). We do **not** use EAS's cloud
build (its free-tier monthly quota kept blocking deploys); kyuhachi is a public
repo, so runner minutes are free.

The app stays **Expo-managed**: `app/ios` is gitignored and regenerated by
`expo prebuild` on every CI run. No native code is committed.

## How a deploy runs

[.github/workflows/deploy.yml](../.github/workflows/deploy.yml), on push to
`master` (or manual `workflow_dispatch`), on `macos-26`:

1. `npm ci` at the repo root (hoisted workspace install).
2. Ruby + bundler (`app/Gemfile`: fastlane + CocoaPods 1.16.2).
3. Decode secrets to files: `GoogleService-Info.plist` (→ `app/`, before
   prebuild), the distribution `.p12`, and the App Store Connect API key JSON
   (→ `$RUNNER_TEMP`).
4. `expo prebuild --platform ios --no-install` — regenerates `app/ios` and runs
   `plugins/withAllowNonModularIncludes` (patches the Podfile), so pods install
   **after** this.
5. `scripts/pod-install.sh` (retries `pod install`).
6. `fastlane ios beta` ([app/fastlane/Fastfile](../app/fastlane/Fastfile)):
   ephemeral keychain → import cert → generate an App Store provisioning profile
   from the ASC API key → set the build number → `build_app` → `upload_to_testflight`.

### Marketing version

`CFBundleShortVersionString` (e.g. `1.0.0`) has a **single source of truth**:
the `version` field in [app/package.json](../app/package.json).
[app.config.js](../app/app.config.js) reads it
(`require("./package.json").version`), so `expo prebuild` -> `Info.plist` ->
fastlane all derive from that one field — there is no second copy to drift.

Bump it in one step (from the repo root):

```sh
npm run version:bump -- patch    # 1.0.0 -> 1.0.1
npm run version:bump -- minor    # 1.0.0 -> 1.1.0
npm run version:bump -- major    # 1.0.0 -> 2.0.0
npm run version:bump -- 1.2.3    # explicit
```

This updates `app/package.json` **and** the `app` entry in `package-lock.json`
(keeping them in sync so `npm ci` doesn't fail in CI). Commit the change;
pushing to `master` triggers the TestFlight deploy. See
[scripts/version.mjs](../scripts/version.mjs).

The `shared` / `functions` package versions are independent internal-package
versions and are deliberately **not** tied to the app's marketing version.

### Build number

`CFBundleVersion` is **not** stored in the repo. It's taken from TestFlight at
build time, scoped to the current marketing version:
`latest_testflight_build_number + 1`, written into `Info.plist`'s
`CFBundleVersion` by fastlane (after prebuild). A failed/non-uploading run
leaves the TestFlight max unchanged, so the next run reuses the same number — no
drift, nothing committed. Bumping the marketing version resets the build number
(no TestFlight builds exist for the new version yet, so it starts at 1).

Because fastlane sets the build number *after* `expo prebuild`, it is not in the
JS-visible config: `Constants.expoConfig.ios.buildNumber` is `undefined` at
runtime. The in-app About screen therefore shows the version only unless it
reads the **native** build number (`expo-application`'s `nativeBuildVersion`,
which reads `Info.plist` directly) — wire that in if the build number must be
displayed.

> EAS Build (`eas.json`, kept as a fallback) manages the build number
> separately via `appVersionSource: "remote"` + `autoIncrement` — that counter
> is independent of the TestFlight-derived number fastlane uses. Treat fastlane
> as the source of truth; don't interleave the two pipelines for the same
> marketing version.

### Signing

A stored **Apple Distribution `.p12`** is imported into an ephemeral keychain;
the **provisioning profile is generated at build time** from the ASC API key
(`get_provisioning_profile`), so there is no long-lived `.mobileprovision` to
store or renew. This requires the ASC key to have **App Manager** (or Admin)
role. If the key is Developer-only, store the profile as a base64 secret and use
the `install_provisioning_profile` fallback noted in the Fastfile.

## Required GitHub secrets

| Secret | What it is |
|---|---|
| `GOOGLE_SERVICE_INFO_PLIST_BASE64` | base64 of `GoogleService-Info.plist` (already set) |
| `IOS_DIST_CERT_P12_BASE64` | base64 of the Apple Distribution certificate `.p12` |
| `IOS_CERTIFICATE_PASSWORD` | the `.p12` export password |
| `ASC_API_KEY_JSON_BASE64` | base64 of `{ "key_id", "issuer_id", "key": "<PEM of .p8>", "in_house": false }` |
| `FASTLANE_APPLE_ID` | _(optional)_ Apple ID, informational with API-key auth |

`EXPO_TOKEN` and `eas.json` are kept as a fallback (and for `eas credentials` /
dev-client builds), but the production deploy no longer uses them.

## One-time setup

1. **Distribution certificate.** Export kyuhachi's existing one as a password-
   protected `.p12`: `cd app && npx eas credentials -p ios` → Production →
   Distribution Certificate. (Or reuse the shared-team cert — Apple team
   `L357YP9W28` is shared with the Kiroku app; certs are per-team.) Don't revoke
   the EAS cert until the new pipeline is verified.
2. **ASC API key.** App Store Connect → Users and Access → Integrations →
   App Store Connect API → generate a key with **App Manager** role. Download
   the `.p8` once; note its `key_id` and the team `issuer_id`. Assemble the JSON
   above (the `key` value is the full PEM contents of the `.p8`).
3. **Add the secrets** (values stay out of logs — the workflow decodes from env):
   ```sh
   base64 -i dist_cert.p12     | gh secret set IOS_DIST_CERT_P12_BASE64 --repo PetrCala/kyuhachi
   base64 -i asc_api_key.json  | gh secret set ASC_API_KEY_JSON_BASE64  --repo PetrCala/kyuhachi
   gh secret set IOS_CERTIFICATE_PASSWORD --repo PetrCala/kyuhachi
   ```

## Verify

- **Locally:** with a real `app/GoogleService-Info.plist` present,
  `cd app && npx expo prebuild --platform ios --no-install && cd ios && pod install`,
  then `xcodebuild -list -workspace Kyuhachi.xcworkspace` (scheme should be
  `Kyuhachi`).
- **In CI:** `gh workflow run deploy.yml --ref <branch>` and watch it to green.
- Confirm the build lands under App Store Connect app `6761064476` → TestFlight.
